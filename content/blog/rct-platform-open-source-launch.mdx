# rct-platform v1.0.4b0: Beta Preview of the Public SDK

The RCT Platform SDK is public on GitHub and currently tracked as a beta-preview checkpoint.

[rct-platform v1.0.4b0](https://github.com/rctlabs/rct-platform) is the current public SDK checkpoint for the open RCT stack. It exposes the [FDIA equation](/protocols/fdia-equation), JITNA Protocol RFC-001, SignedAI multi-LLM consensus, the Delta Engine, and five reference microservices under Apache 2.0. The current public proof lane is **1,287 passing tests at 92% coverage**. Broader enterprise runtime evidence lives in a separate private snapshot and should not be treated as GitHub-verifiable proof.

This is not a mock package. It is the public SDK layer used to verify the open algorithms, CLI flows, and reference services. The production enterprise runtime remains a separate operational layer.

## What ships in v1.0.4b0

The public release contains everything needed to understand, run, and extend the core RCT algorithms:

| Component | Description |
|---|---|
| **FDIA Scorer** | Full equation engine — `core/fdia/fdia.py` |
| **JITNA RFC-001** | Intent protocol implementation — `rct_control_plane/` (15 modules) |
| **SignedAI** | Multi-LLM consensus layer — `signedai/core/` |
| **Delta Engine** | 74% memory compression via state diffing — `core/delta_engine/` |
| **Regional Adapter** | 8 language-pair support — `core/regional_adapter/` |
| **5 Microservices** | Reference implementations validated in the public SDK suite |
| **CLI** | `rct compile`, `rct governance`, `rct timeline`, `rct doctor`, `rct start --ui-test` |
| **Evidence** | Public SDK verified at 1,287 tests / 92% coverage; benchmark and enterprise evidence are tracked separately |

The current public checkpoint is **1,287 passed, 0 skipped, 0 failed** across Python 3.10, 3.11, and 3.12, with **92%** coverage. No Docker is required to run the test suite, the offline demo, or the CLI onboarding path.

## Getting started (5 minutes, no API keys)

```bash
# Clone and install
git clone https://github.com/rctlabs/rct-platform.git
cd rct-platform
pip install -e .

# Run the offline demo
python examples/quickdemo.py

# Run the FDIA benchmark
python benchmark/fdia_benchmark.py --verbose

# Try the CLI
rct compile 'Protect resources from hostile agents'
rct governance
```

Expected output from `quickdemo.py`:

```

Consensus: PASSED ✅ (threshold: 4/6)
Final decision: EXECUTE — all 7 gates cleared, FDIA ≥ threshold, intent signed
Memory compression ratio: 5.0× (baseline + 4 deltas vs 5 full-state copies)
```

## Why open source, and why now

The [RCT Open Protocol](/open-protocol) has always been designed to be verifiable. Constitutional AI that cannot be inspected is not constitutional — it is marketing. Publishing the reference implementation is the minimum required to make the governance claims credible.

Three specific reasons drive the current beta-preview positioning:

**1. The public SDK now has a canonical proof source.** Test count and coverage claims are tracked from `docs/testing/TESTING_CANONICAL.md`, which reduces drift between README, roadmap, release notes, and launch copy.

**2. The CLI onboarding path is materially stronger.** `rct init`, `rct doctor`, and `rct start --ui-test` make it possible to verify the local developer experience without API keys before moving into live integrations.

**3. Evidence lanes are clearer.** Public SDK validation, benchmark framing, and enterprise-private runtime claims are now being separated explicitly instead of blended into one marketing number.

## What the Apache 2.0 license means for you

Apache 2.0 is a permissive license with patent protection. In practice:

- **Commercial use is allowed.** You can use, modify, and distribute `rct-platform` in proprietary products without publishing your modifications.
- **Attribution is required.** Include the `NOTICE` file and the copyright header in any distribution.
- **Patent grant included.** Contributors grant you a royalty-free, worldwide patent license covering their contributions.
- **No copyleft.** Unlike GPL, Apache 2.0 does not require derivative works to be open source.

The enterprise layer — 62 runtime components, the HexaCore inference cluster, RCTDB, and the MEE production pipeline — remains proprietary. When those surfaces are referenced publicly, they should be labeled as an enterprise-private snapshot rather than GitHub-verifiable proof.

## What the beta-preview wave adds

The current wave is centered on first-run clarity and public evidence hygiene. Key additions include:

- **CLI diagnostics and first-run UX** — `rct doctor`, better `rct start`, log/status commands, and a wheel-safe `rct init`
- **Canonical claim sync** — public docs and distribution materials aligned to the current checkpoint
- **Cleaner scope boundaries** — public SDK proof separated from benchmark scope and enterprise-private runtime snapshots

## Relationship to the enterprise layer

The SDK and the enterprise layer have a deliberate interface boundary:

```
rct-platform (Apache 2.0, public)
    └── core/          ← FDIA, Delta Engine, Regional Adapter
    └── signedai/      ← Multi-LLM consensus
    └── rct_control_plane/  ← JITNA RFC-001
    └── microservices/ ← 5 reference implementations

RCT Enterprise (proprietary, rctlabs.co)
  └── 62 runtime components
    └── RCTDB (8-dimensional memory schema)
    └── HexaCore 7-model inference cluster
    └── MEE production pipeline
    └── 4,849 enterprise-private test snapshot
```

The SDK is not a stripped-down version of the enterprise layer. It is the canonical reference implementation of the protocols and algorithms that the enterprise layer uses. They share a common interface contract — the JITNA RFC-001 spec — but they are built and maintained independently.
