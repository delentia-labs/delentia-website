# rct-platform v1.0.4b0: Beta Preview ของ Public SDK

RCT Platform SDK เปิดสาธารณะบน GitHub แล้วและตอนนี้อยู่ใน checkpoint แบบ beta preview

[rct-platform v1.0.4b0](https://github.com/rctlabs/rct-platform) คือ checkpoint ปัจจุบันของ public SDK สำหรับ open RCT stack มันเปิดเผยสมการ FDIA, JITNA Protocol RFC-001, SignedAI multi-LLM consensus, Delta Engine และ microservice อ้างอิง 5 รายการภายใต้ Apache 2.0 โดย proof lane สาธารณะปัจจุบันคือ **1,287 tests ที่ผ่านทั้งหมด และ 92% coverage** ส่วน evidence ของ enterprise runtime ที่กว้างกว่านี้ถูกแยกเป็น private snapshot และไม่ควรถูกใช้เป็น proof ของ GitHub repo

นี่ไม่ใช่แพ็กเกจตัวอย่าง มันคือ public SDK layer ที่ใช้ยืนยัน algorithm, CLI flow และ reference services ของฝั่งเปิดเผย ส่วน runtime production เต็มรูปแบบยังเป็นอีกชั้นหนึ่ง

## สิ่งที่มาใน v1.0.4b0

| Component | คำอธิบาย |
|---|---|
| **FDIA Scorer** | Engine ของสมการแบบเต็ม — `core/fdia/fdia.py` |
| **JITNA RFC-001** | Intent protocol implementation — `rct_control_plane/` (15 modules) |
| **SignedAI** | Multi-LLM consensus layer — `signedai/core/` |
| **Delta Engine** | Memory compression 74% ผ่าน state diffing — `core/delta_engine/` |
| **Regional Adapter** | รองรับ language-pair 8 คู่ — `core/regional_adapter/` |
| **5 Microservices** | Reference implementations ที่อยู่ใน public SDK suite |
| **CLI** | `rct compile`, `rct governance`, `rct timeline`, `rct doctor`, `rct start --ui-test` |
| **Evidence** | Public SDK verified ที่ 1,287 tests / 92% coverage โดยแยกจาก benchmark และ enterprise snapshot |

checkpoint สาธารณะปัจจุบันคือ **1,287 passed, 0 skipped, 0 failed** บน Python 3.10, 3.11 และ 3.12 พร้อม **92%** coverage โดยไม่ต้องใช้ Docker ในการรัน test suite, offline demo หรือเส้นทาง onboarding ของ CLI

## เริ่มต้นใช้งาน (5 นาที ไม่ต้องใช้ API key)

```bash
# Clone และติดตั้ง
git clone https://github.com/rctlabs/rct-platform.git
cd rct-platform
pip install -e .

# รัน offline demo
python examples/quickdemo.py

# รัน FDIA benchmark
python benchmark/fdia_benchmark.py --verbose

# ลอง CLI
rct compile 'Protect resources from hostile agents'
rct governance
```

Output ที่คาดไว้จาก `quickdemo.py`:

```
✅  Best action: act_001 (cooperate) — highest FDIA score
Consensus: PASSED ✅ (threshold: 4/6)
Final decision: EXECUTE — all 7 gates cleared, FDIA ≥ threshold, intent signed
Memory compression ratio: 5.0× (baseline + 4 deltas vs 5 full-state copies)
```

Output ที่คาดไว้จาก `fdia_benchmark.py`:

```
RCT FDIA Score       : 0.9167
Industry Baseline    : 0.6500
Delta vs Baseline    : +0.2667 (+41.0%)
✅  Benchmark PASSED
```

## ทำไมถึง open source และทำไมตอนนี้

[RCT Open Protocol](/open-protocol) ถูกออกแบบมาให้ verify ได้เสมอ Constitutional AI ที่ไม่สามารถตรวจสอบได้ไม่ใช่ constitutional — มันคือ marketing การ publish reference implementation คือขั้นต่ำที่จำเป็นเพื่อให้ governance claims มีความน่าเชื่อถือ

เหตุผลสามประการที่กำหนดสถานะ beta preview ปัจจุบัน:

**1. Public SDK มี canonical proof source แล้ว** ตัวเลข test count และ coverage ถูกยึดจาก `docs/testing/TESTING_CANONICAL.md` ซึ่งลด drift ระหว่าง README, roadmap, release note และ launch copy

**2. CLI onboarding แข็งแรงขึ้นจริง** `rct init`, `rct doctor` และ `rct start --ui-test` ทำให้ตรวจสอบ first-run experience ได้โดยไม่ต้องมี API key ก่อนเข้าสู่ live integration

**3. Evidence lanes ชัดขึ้น** public SDK validation, benchmark framing และ enterprise-private runtime claims ถูกแยกจากกันชัดกว่าการใช้ตัวเลขก้อนเดียวแบบเดิม

## Apache 2.0 หมายความว่าอะไรสำหรับคุณ

- **ใช้เชิงพาณิชย์ได้** คุณสามารถใช้ แก้ไข และ distribute `rct-platform` ในผลิตภัณฑ์ proprietary โดยไม่ต้องเผยแพร่การแก้ไขของคุณ
- **ต้องให้ attribution** รวมไฟล์ `NOTICE` และ copyright header ในทุก distribution
- **มี patent grant** Contributors ให้ royalty-free, worldwide patent license ครอบคลุม contributions ของพวกเขา
- **ไม่มี copyleft** ต่างจาก GPL Apache 2.0 ไม่กำหนดให้ derivative works เป็น open source

Enterprise layer — 62 runtime components, HexaCore inference cluster, RCTDB และ MEE production pipeline — ยังเป็น proprietary เมื่ออ้างถึง publicly ควรติดป้ายว่าเป็น enterprise-private snapshot ไม่ใช่ GitHub-verifiable proof

## รายละเอียด Public Proof

| Dimension | จำนวน | หมายเหตุ |
|---|---|---|
| Public SDK suite | 1,287 | checkpoint เต็มของ public repository |
| Direct microservice slice | 297 | reference microservices โดยตรง |
| Benchmark suite | — | Script แยกใน `benchmark/` ไม่ใช่ใน pytest |

Coverage ปัจจุบันรายงานที่ **92%** ใน canonical checkpoint โดย source สาธารณะที่ authoritative คือ `docs/testing/TESTING_CANONICAL.md` ใน repository ถ้าต้องการตัวเลขล่าสุดให้ยึดไฟล์นั้นแทน snapshot เก่าของบทความ

## สิ่งที่ wave beta preview นี้เพิ่มเข้ามา

- **CLI diagnostics และ first-run UX** — `rct doctor`, `rct start` ที่ดีขึ้น, คำสั่ง log/status และ `rct init` ที่ปลอดภัยกับ wheel install
- **Canonical claim sync** — เอกสารสาธารณะและ distribution materials ถูก sync กับ checkpoint ปัจจุบัน
- **Scope boundary ที่สะอาดขึ้น** — public SDK proof แยกจาก benchmark scope และ enterprise-private snapshot

## ความสัมพันธ์กับ Enterprise Layer

```
rct-platform (Apache 2.0, public)
    └── core/          ← FDIA, Delta Engine, Regional Adapter
    └── signedai/      ← Multi-LLM consensus
    └── rct_control_plane/  ← JITNA RFC-001
    └── microservices/ ← 5 reference implementations

RCT Enterprise (proprietary, rctlabs.co)
  └── 62 runtime components
    └── RCTDB (8-dimensional memory schema)
    └── HexaCore 7-model inference cluster
    └── MEE production pipeline
    └── 4,849 enterprise-private test snapshot
```

SDK ไม่ใช่ stripped-down version ของ enterprise layer มันคือ canonical reference implementation ของ protocols และ algorithms ที่ enterprise layer ใช้ พวกเขา share interface contract ร่วมกัน — JITNA RFC-001 spec — แต่สร้างและดูแลรักษาแยกกัน
